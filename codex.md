# 一、项目目标

我要做的不是“机场中转转发系统”，而是一个**管理面系统**：

1. 服务器端继续使用现有的 **WireGuard 原生节点**
2. 客户端侧主要使用 **Mihomo / Clash.Meta**
3. 通过 **订阅链接** 下发配置
4. 每个用户有**专属订阅链接**
5. 每个用户可以看到自己的**每日流量使用情况**
6. **不要让用户流量经过统一中转层**
7. **不要为了统计而把 VPN 数据面改成转发架构**
8. 统计方式应基于 **WireGuard peer 级别流量计数**
9. 我需要的是“像机场一样的管理体验”，但**底层数据面仍然是客户端直连三台 WireGuard 节点**

也就是说：

- **订阅鉴权**放在管理面
- **流量统计**放在管理面
- **VPN 数据流**仍然直接走 WireGuard 节点
- 不要把这套东西做成会明显拖慢 VPN 性能的“统一中转代理架构”

# 四、我要实现的核心业务能力

## 1. 用户系统

每个用户应该至少有：

例如形态可以类似：

- 或 `https://sub.wfqc8.cn/sub/<token>/nodes.yaml`

也就是说：

- 一个用户在 206 上有一条独立 peer
- 一个用户在 100 上有一条独立 peer
- 一个用户在 101 上有一条独立 peer

## 4. 流量统计

- 每个用户每天使用了多少流量

- 近似“机场后台”的用户视角

注意：

- 统计必须基于 **WireGuard peer 的 transfer-rx / transfer-tx**
- 不要让 VPN 流量经过统一转发器
- 允许通过定时任务采集 `wg show` / `wg show all dump` 结果入库
- 允许通过 SSH 到 100/101 拉数据
- 允许通过本机读取 206 的数据
- 可以按分钟采样，然后按天汇总

# 六、我对技术方案的明确要求

## 1. 不走统一转发

这是非常关键的要求：

- 不要把这套系统做成“所有用户流量必须经过统一中转服务”
- 不要为了鉴权或统计，额外让数据流走用户态转发网关
- 我更看重 WireGuard 原本的性能和稳定性

## 2. 订阅鉴权与数据面分离

请把：

- 订阅鉴权
- 用户鉴权
- 配置生成
- 流量统计
- 用户管理

全部视为“管理面”

请把：

- WireGuard 隧道转发

视为“数据面”

两者要明确分离。

## 3. 每用户独立 peer

这是统计和停用的基础，必须实现。

## 4. 支持停用用户

当用户被停用、过期、超额时，至少应支持其中一种或多种方式：

- 订阅返回空节点
- 订阅不再返回该用户节点
- 自动从三台服务端移除 / 禁用 peer
- 后台标记失效

## 5. 优先做最小可用版本

我现在并不追求一次性做成完整商业机场面板。

请你先按 **MVP 最小闭环** 来设计和编码，优先做到：

1. 能创建用户
2. 能为用户生成三台节点的 peer
3. 能生成用户专属 `main.yaml` 和 `nodes.yaml`
4. Clash/Mihomo 可实际使用
5. 能采集 peer 流量并做每日统计
6. 能查看某个用户的流量情况

## 6. 技术栈建议

请优先考虑：

- Python 3
- FastAPI
- SQLAlchemy
- Alembic
- Pydantic
- SQLite（开发）/ PostgreSQL（生产可切换）
- APScheduler 或 cron 方案用于流量采集
- Paramiko 或系统 SSH 调用，用于从 206 拉取 100/101 的 `wg show all dump`

如果你认为更合理，也可以提出优化，但不要偏离“轻量、可维护、易部署”的目标。
